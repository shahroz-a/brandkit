# BrandKit OS

BrandKit OS is an open-source developer branding toolkit. It generates reproducible brand tokens and deterministic assets for apps, documentation, repositories, social previews, favicons, manifests, and launch materials.

The project is local-first by design: no backend, no auth, no database, no paid APIs.

## What Is Included

- Visual web editor built with Next.js App Router, TypeScript, TailwindCSS, shadcn-style primitives, Zustand, and deterministic SVG rendering.
- Core package for brand tokens, validation, template execution, and asset generation.
- CLI with `init`, `generate`, `export`, `build`, `sync`, `doctor`, and `validate` commands.
- SDK and React packages for framework and component integrations.
- Optional uploaded logos, generated logo styles, premium meta-image layouts, gradients, grids, and patterns.
- LLM/MCP tool manifest export for teams that want to host their own agent-facing generation service.
- Template package with a small authoring API and production-ready starter templates.
- Documentation, governance files, issue templates, CI, and roadmap.

## Quick Start

```bash
npm install
npm run dev
```

Open the local URL printed by Next.js. The app stores kits in the browser and exports assets locally.

The repository also includes `pnpm-workspace.yaml` for pnpm-based contributors, while npm remains the default package manager for CI and Vercel:

```bash
corepack enable
pnpm install
pnpm --filter @brandkit/web dev
```

## Deployment

### Vercel

Use the default root build command:

```bash
npm run build
```

### Cloudflare Pages

BrandKit OS is a static Next.js app on Cloudflare Pages. Use these build settings from the repository root:

```text
Build command: npm run build:cloudflare
Build output directory: apps/web/out
Deploy command: leave blank
```

Cloudflare Pages sets `CF_PAGES=1`, and Cloudflare Workers Builds sets `WORKERS_CI=1`, so the default `npm run build` will emit the same static export during Cloudflare builds. If a deploy command is configured for Pages, do not use `npx wrangler deploy` from the workspace root; Pages should upload `apps/web/out` directly.

For Cloudflare Workers Builds that run `npx wrangler deploy`, the checked-in `wrangler.toml` and `apps/web/worker.ts` serve the same static export through Workers static assets.

## CLI

```bash
npm run brandkit -- init
npm run brandkit -- generate
npm run brandkit -- doctor
npm run brandkit -- mcp
```

The CLI reads `brand.json` and writes deterministic assets to `brandkit-assets/` or `public/` for web projects.
`brandkit mcp` prints a JSON tool manifest for self-hosted MCP or LLM adapters.

## Brand Tokens

```json
{
  "name": "BrandKit",
  "description": "Open-source developer branding toolkit.",
  "font": "Inter",
  "radius": 24,
  "primary": "#2563EB",
  "secondary": "#111827",
  "accent": "#F97316",
  "logoSource": "generated",
  "logoShape": "squircle",
  "logoStyle": "spark",
  "metaGradient": "brand",
  "metaPattern": "grid",
  "metaPatternScale": 36,
  "metaIntensity": 68,
  "metaLayout": "classic"
}
```

Everything generated by BrandKit OS derives from this token file.
Uploaded logos are stored as optional image data URIs in `brand.json`; keep them small if you plan to commit or share the file.

## Repository Map

- `apps/web` - local-first visual editor and exporter.
- `packages/core` - token engine, validators, SVG renderers, asset manifest.
- `packages/cli` - Node CLI for scanning, validating, generating, exporting, and syncing.
- `packages/sdk` - framework-agnostic functions.
- `packages/react` - reusable preview components.
- `packages/templates` - template authoring primitives and included templates.
- `docs` - contributor and integration guides.

## Use Cases

- Generate premium OpenGraph, Twitter, Product Hunt, blog, newsletter, GitHub, and YouTube social images.
- Export generated or uploaded logo systems, favicons, manifests, metadata, and robots files.
- Validate brand tokens in CI with `brandkit doctor` or `brandkit validate`.
- Run batch generation from CSV, TSV, or JSON.
- Expose `@brandkit/core` or the `brandkit mcp` manifest behind your own hosted MCP/HTTP adapter for LLM workflows.

BrandKit OS does not ship a hosted backend. Teams that need shared generation endpoints, queues, auth, storage, or MCP servers should self-host those pieces around the core package.

## Principles

Branding should be version controlled, reproducible, generated from code, deterministic, and automatable.
